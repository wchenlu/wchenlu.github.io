TCP/IP参考模型

为什么提出这个模型

因为OSI的协议比较复杂，且tcp和IP协议被广泛应用，所以出现了该模型成为网络的主流模型的原因。

该模型其实就是把OSI模型进行了一个集合：

tcp/IP协议模型如下：

为什么说他对OSI模型进行了一个集合？

因为它应用层把SI模型的应用层、表示层、会话层全部集合在里面，OSI模型的传输层对应它主机到主机层，网络层对应它的因特网层，最后的数据链路层和物理层就归纳为它的网络接入层。

两者的区别？

这里不去过多纠结他们之间的细微差距简单总结就是tcp/ip模型适合用来生产而OSI模型适合用来学习。

和之前的模型一样，会介绍一下模型里不同层不同作用，不过因为比较相似，重点在于介绍协议：

协议？什么是协议？ 可以简单理解为网络工作过程中的一些规则

一、应用层

第五层应用层协议：

应用层协议名称 使用的端口 使用TCP还是UDP 协议作用
HTTP 80 TCP 超文本传输协议、提供浏览网页的服务
Telnet 23 TCP 远程登录协议、远程管理服务
FTP 20和21 TCP 文件传输协议，提供互联网资源共享
SMTP 25 TCP 简单邮件传输协议
TFTP 69 UDP 简单文件传输协议
应用层是封装在传输层的所以会用到传输层的协议：TCP和IP

TCP协议和UDP协议的区别是tcp提供的是可靠传输，保证数据的完整和准确，udp速度快，但是可能会丢失一些数据

为了区分数据流量是来自哪个协议，所以会有端口号来标识它。

常用应用层协议介绍：

说起常用，作为文件共享的FTP位当首位：

FTP协议：

作用是传输文件，它在传输时使用TCP协议。就是把一个文件发到另外一个电脑，这个电脑可以是服务器也可以客户机。这种一个客户端到服务器端的叫C/S模式，这种模式下的客服端在FTP连接后可以对FTP服务器上的文件进行操作。而FTP服务器是运行FTP服务的设备。

Telnet协议：

数据网络的远程登录协议，这个协议为用户提供在本地操作远程设备工作的能力，以后会解释怎么使用。这个协议也是客服端服务器，不管大多情况的客户端是电脑，服务器就是一些网络设备，例如服务器、防火墙、交换机等。

HTTP协议：

从应用方面来说，这个协议最为广泛，它提供了一种发布和接收特殊页面的功能，值得一提的是，这个协议设计的最初目的是为了提供一种接收和发布HTML页面的方法。自然这个协议也使用C/S，工作过程都很收悉，在客户端上输入一个需要访问的地址，这个地址发到服务器上，服务器返回一个页面。

二、传输层

传输层的协议会接收来自应用层的协议，对其封装上传输层的头部，并建立一个“端到端的连接”。加孵传输层头部信息后，数据称为段（Segment）。也就是PDU（协议数据单元）。

既然传输层会为数据贴上标签，那这些标签又有哪些信息呢？

TCP和UDP会给数据添加字段，TCP会加上二十个字节的字段，UDP会加上八个字节的字段，TCP的字段虽然多，但它可以通过这些字段来保证传输，比如数据丢失可以再传一次，数据传输错误，被人篡改也是可以查出来的 ；有一点需要注意，不管是UDP还是TCP他们都有一个源端口和目的端口，需要注意的是端口号！

！！！！端口号其实会区分应用！！！

端口还有一点需要注意：

一个IP地址可以有端口的长度为16比特，也就是说一个IP地址最多可以有二的十六次方减一个端口，也就是65536个端口

为了方便理解这里介绍一下TCP和UDP端口号的工作方式：

比如客户在客户端上浏览百度，那么应用层就会选择提供网页的服务协议HTTP，这个协议到了传输层之后会选择一个TCP协议的端口号，这个端口会有一个源端口和目的端口；选择的源端口会区分应用。然后在到达服务器后，会选择80端口，因为这个端口对应的就是HTTP协议。

具体为：客服端封装一个信息发出去，这个封装的信息包括：
IP头部：包含了源IP和目的IP
TCP头部：源的端口号和目的端口号
HTTP载荷：这里主要是一些语句

以上的端口需要注意一下，客户端使用的源端口一般随机分配，目标端口则由服务器的应用指定；

源端口号一般是没有使用的，且大于1023；

传输层的重点毫无疑问是TCP和UDP协议：

之前已经简略说过两者的差别，接下来对两者进行详细的学习记录：

TCP：

它是一种面向连接的，及其可靠的传输层通信协议，为什么说他极其可靠？因为他保证了传输数据的万无一失。

为什么说它万无一失，这里提一下它的几种机制：

建立连接——三次握手：

什么是三次握手？可以理解为在数据开始通信前确保发送和接收两方的连接已经建立好了。

记录的三次握手如下：

假设PC1要把数据发送给pc2

三次握手的第一次，PC1会发送一个信息，这个信息包括了IP头部和TCP头部，IP头部包括的依然是源和目的的IP地址，而TCP头部会包括一个seq和一个ACK还有一个flags，这个tcp头部就是告诉目的地设备：接下来要和你通信，然后我通信的编码是从哪开始的，（具体从哪开始就是之前那个seq的值，这个值一般是数字），ACK就是我要确认的东西，因为这是第一条，所以一般情况下为零。而flags是一个SYN置位，这个一般就是0和1两种取值。然后这次信息发送之后会告诉目的地，我接下开始通信，你要给我留足够的资源。

三次握手的第二次，PC2 在收到这条信息后会发回去一条信息，这条信息和PC1发送时包括的信息一样，不过注意，这里的PC2 作为发送方，会把之前的PC1的目的地址和源地址反过来。还有就是PC2的序列号（就是之前的seq）是另外一个值，和之前PC1的没有关系。而ACK作为确认信息自然是有关系的，ACK等于之前PC1发送的值加一，这个加一的含义就是告诉PC1，发送的信息收到了，可以开始发送下一条了。而flags会把SYN同时标注为一。

三次握手的第三次，PC1会在接到PC2的回应后发一条信息告诉PC2，信息收到了，然后这条信息的seq就是之前第一次握手时PC发出的值加一（也就是标识这是PC1第二次发送的意思），ACK是PC2发出的信息里的seq加一。

这三次握手成功结束后代表PCT连接建立，可以开始传输了。建立之后就开始传输：

TCP会使用序列号和序列号字段来实现数据的可靠和有序传输。

PC1发送时会有很多“包裹”需要发送，在发送时，和之前一样，有IP头部和TCP头部以及载荷，载荷会告诉PC2发送的长度

PC自然会发回一条消息，告诉PC1接收情况，这里的PC2返回信息里TCP头部的序列号和PC一一样会加一，不过注意一下，序列号都是自己的，比如PC1的seq是a，PC2的seq是b那么每次发送都会在自己的seq上加一，而ACK就是确认对方的信息，所以PC2的信息发送的**ACK等于a+1+x（x是PC1发送数据的长度），注意PC2的载荷是零，因为PC2只是确认收到做一个回应并没有数据发给PC1。

PC1在收到这个信息后，确认PC2收到，然后就会发送第二条，这一条需要注意的是seq=a+（y+1），y+1的原因是之前的y都已经发送过去了 ，ACK自然是b+1。自然PC2会做出回应。

那么问题来了，如果发生数据丢失怎么办，怎么发现？

在发送过程中，接收方在一定时间内没有收到，那么就会认为数据丢失。

TCP的窗口滑动机制：

这个机制是TCP用来控制传输速率的

有些情况下，接收方和发送方能处理数据的大小可能不一样，比如发送方有十个报文要发送，可接收方只能接收八个，那么就会导致有两个报文被丢弃，这个时候就需要这个窗口滑动机制了。

这个机制会在建立连接的三次握手时，发送一个win字段，双方确认可以发送的数据大小数量，比如接收方在握手时会告诉接收方我要发送十个报文，那么接收方会在返回的确认信息里通过win字段告诉发送方能接收的数量。假设第一次传输之后，把接收方的处理占满，那么接收方在返回信息的时候就会告诉发送方win=1，以免造成数据的丢失。

有一点需要注意，win字段只表示可以接收的大小，如果处理接收方的少发一点的请求时，会载荷接收方能接收的数据大小，而发送方win的大小是不受影响的

TCP四次挥手断开连接：

当数据传输完成后，TCP通过四次挥手来断开TCP连接，释放系统资源。

第一次挥手发送一个FIN请求断开连接，发送的信息里在TCP头部会包含一个FIN置位，就是告诉对方这是断开连接请求。第二次挥手，接收到这个信息后，会回应一个消息，也是包括IP头部和TCP头部，这个消息发送后，两者的的单向连接就会断开，但是TCP提供的是双向连接所以接收到这个消息后会返回一个FIN请求断开连接，这就是第三次挥手然后TCP的规则就是你给我发，我就会回应，所以就有了第四次挥手自此TCP连接断开。

以上就是TCP用来保证传输可靠的机制。

三、网络层和数据链路层

1、网络层

传输层建立主机进程与进程之间的连接，而网络层负责一台主机到一台主机的数据传递

网络层的PDU（协议数据单元）称为 PacKet（包）；网络层又称为因特层。

作用：

1.为网络设备提供逻辑地址

2.负责数据包的寻找路径和转发

常见的协议有：IPv4、IPv6、ICMP、IGMP等。

网络层协议工作过程：

这一层一样会对数据进行封装，在传输层接到数据后它会添加一个IP报文头部，也就包含源地址和目的地址。

基于网络层地址的包转发过程：

发送端会先进行上述的封装，然后会发给连接的路由器，路由器根据路由表选择最合适的端口路线进行转发

2.数据链路层

位于网络层和物理层之间，向网络的IP等协议提供服务，数据链路层的PDU（协议数据单元）称为Frame（帧）。

这一层也有许多协议，其中以太网协议是数据链路层最常见的协议，这个协议通常用在局域网当中。

数据链路层位于网络层和物理层之间：

1.数据链路层向网络提供“段内通信”；

2.负责组织帧、物理编址、差错控制等功能。

常见的数据链路协议有

以太网协议、PPPo、PPP等

以太网与MAC地址

以太网是一种广播式数据链路协议，支持多点接入。

个人电脑遵循的就是以太网标准

*一般情况下，一个广播域对应一个IP网段。

每台主机都有一个固定的Mac地址，一般情况不可更改。MAc地址可以叫以太网地址，也可以叫物理地址。

MAC地址用于在一个IP网段里寻找具体的物理设备，工作在数据链路层的设备–比如以太网交换机，会维护一张Mac地址表，用于指导数据转发。

交换机维护的是Mac地址表，路由器维护的是Mac地址表。

ARP协议

address Resoltion Protocol（地址解析协议）：根据IP地址解析获得其对应的Mac地址。

ARP协议的工作过程

在通信中，一端向另外一端发送一个ARP请求报文，这个报文会包括IP头部，和MAc地址，其中目的Mac地址并不知道

详细解释理解为：

PC1 与PC2进行通信，使用ping命令，则会发出一个ICMP的请求包，而作为报文发出，自然进行封装，这里先提两层，网络层和链路层，它会在网络层获得一个源IP地址和一个目的IP地址，会在链路层获得一个自身的Mac地址，可并不知道到目的地Mac地址是多少，而没有目的Mac地址封装不能成功。

所以怎么拿到Mac地址？

这个时候就需要ARP协议，它会发送一个ARP请求，这个请求自然也要封装，不过它不知道目的Mac地址，所以它会在封装时填入一个全F的Mac地址，这个会实现一个广播的功能，会让一个广播域下的所有主机都收到，然后目的主机收到后就会个源主机返回一个信息，这个信息包括了它的Mac地址，有了这个地址后，封装就可以完成。

ARP的请求报文会直接封装在以太网协议里，所以不属于三层的包。

ARP协议的工作原理

主机会有一个ARP表，这个表包括IP地址和Mac地址，如果主机第一次开机，那么这个表其实是空的。所以ARP的第一步会看自身的缓存，如果这个时候是空的会通过ARP去获取，如果有目的Mac地址，会直接拿来封装数据帧。如果是空的，就有了第二步发送一个ARP请求，通过发送一个ARP Request（请求）报文来获取目的Mac地址，因为不知道目的Mac地址，所以会在目的Mac地址封装为零。所有的主机在收到这个广播消息后，会做两件事：检查这个广播消息的目的IP地址和自己的IP地址是否匹配，如果匹配把这个地址记录到自己的ARP缓存表。第二件事情就是第三步ARP的响应响应就是目的主机回应源主机一个包，这个包叫ARP Reply（回应），这个回应就会携带目的主机的Mac地址，这里的目的Mac地址会是源主机，不会是全F了。最后就是第四步返回源主机后，源主机缓存下这个Mac地址，因为来源是ARP的报文，所以他是动态的。

四、物理层

这一层之前的数据以经封装完成，数据到达物理层后，会根据介质的不同，把数字信号转换成对应的信号：

介质 对应信号
光纤 光信号
双绞线 电信号
无线网 电磁波

物理层的PDU（协议数据单元）是比特流（Bitstream）。
